# GROMACS Cluster Execution Playbook

## Purpose

Use this reference when a GROMACS workflow is moving from preprocessing into scheduled cluster execution.

## Pairing rule

Use this together with `hpc-orchestration`.

- this file owns GROMACS-specific stage sequencing and continuation logic
- `hpc-orchestration` owns scheduler, transfer, storage, and monitoring scaffolds

## Preflight

Before a production `mdrun`:

1. confirm the `.tpr` file was generated by a clean `grompp` run
2. confirm the expected topology, coordinate, and index inputs are version-consistent
3. confirm output and checkpoint paths are writable
4. decide whether the stage is EM, NVT, NPT, or production
5. size the job from one short representative benchmark

Do not debug topology issues inside a large production allocation.

## Stage-aware execution

Typical GROMACS lifecycle:

1. preprocessing with `grompp`
2. stage-specific `mdrun`
3. checkpoint-aware continuation if needed
4. post-processing or the next `.tpr` generation step

Keep each stage in a separate, clearly named run directory or at least with distinct output prefixes.

## Launch strategy

Portable starting point:

- let the scheduler allocate resources
- launch `mdrun` through `srun` unless the site packaging says otherwise
- make thread and rank choices explicit rather than relying on hidden defaults

Example shape:

```bash
export OMP_NUM_THREADS=4
srun -n 8 gmx_mpi mdrun -deffnm md
```

If the site build is thread-only rather than MPI-enabled, adapt the launcher to that actual binary.

## Logs worth watching

High-value checks:

- `grompp` warnings that were forced through casually
- LINCS warnings
- domain decomposition failures
- unstable pressure or temperature during early equilibration
- checkpoint creation and append behavior

## Storage and output

GROMACS outputs can become large quickly.

Recommended habits:

- keep per-stage output prefixes distinct
- control trajectory output frequency intentionally
- keep checkpoints on durable enough storage for continuation
- stage heavy trajectory data back only when the analysis plan requires it

## Restart and continuation

Before continuation:

- confirm the checkpoint file corresponds to the intended `.tpr`
- decide whether to append or continue with a new prefix
- keep one clear naming policy for stage restarts

Do not mix continuation modes casually across long jobs.

## Failure patterns

| Symptom | Likely cause | First repair |
| --- | --- | --- |
| production job starts from a bad setup | `grompp` warnings were ignored | return to preprocessing and rebuild the `.tpr` |
| batch run is unstable early | equilibration or coupling settings are too aggressive | step back to the prior stage and retune the `.mdp` |
| continuation creates confusing files | append policy or prefix naming drifted | standardize `-deffnm`, checkpoint, and continuation naming |
| performance is poor at scale | rank-thread layout is wrong for the site build | benchmark a smaller geometry and adjust MPI or thread counts |
